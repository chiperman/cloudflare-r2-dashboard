# 认证重置流程修复变更汇总

为了彻底修复“通过重置密码链接可非法访问系统”的安全漏洞，并优化相关用户体验，我们对以下文件进行了关键修改。

## 1. Middleware (核心安全拦截)

**文件路径**: `src/middleware.ts` 
*(原位于 `middleware.ts`，现已按 Next.js 规范移入 `src/`)*

- **做了什么修改？**
  - **位置移动**：将中间件移入 `src` 目录。
  - **JWT 深度解析**：添加了手动解码 Access Token JWT 的逻辑，直接提取 `amr` (Authentication Method Reference) 字段。
  - **强制路由重定向**：
    - 如果检测到 `recovery` (重置模式)：强制跳转到 `/auth/reset-password`，禁止访问首页或 API。
    - 如果是普通用户且没有重置 Code：禁止直接访问重置页面。
    - **清理日志**：移除不必要的 `console.log`，只保留关键错误。

- **为什么要修改？**
  - **位置问题**：Next.js 在有 `src` 目录时会忽略根目录的中间件，导致之前中间件根本没生效。
  - **状态检测问题**：Supabase 的 `getUser()` 在 PKCE 模式下有延迟，经常拿不到即时的“重置状态”。直接解析 JWT 是唯一能 100% 捕获该状态的手段。

- **如果不修改会怎样？**
  - **严重漏洞**：用户点击重置邮件后，可以直接去首页浏览敏感文件，而不需要修改密码。

## 2. AuthDialogManager (客户端 UI 逻辑)

**文件路径**: `src/components/auth-dialog-manager.tsx`

- **做了什么修改？**
  - **引入 Hook**：使用了新封装的 `useRecoveryMode` Hook。
  - **状态限制**：当检测到是 `recovery` 模式时：
    - **隐藏**：个人资料弹窗 (ProfileDialog)。
    - **显示**：仅显示用户名和退出按钮。
    - **移动端优化**：在移动端隐藏内联的退出按钮（避免和菜单里的退出重复）。

- **为什么要修改？**
  - **提升体验**：防止用户在“只能改密码”的状态下看到“编辑资料”按钮，点击了也会被后端拦截，体验很差。
  - **消除歧义**：明确告知用户当前状态受限。

- **如果不修改会怎样？**
  - 功能虽然安全（后端已拦截），但前端会显示不可用的按钮，用户点击后会被莫名弹回，产生困惑。

## 3. Header (顶部导航栏)

**文件路径**: `src/components/header.tsx`

- **做了什么修改？**
  - **最终状态**：可以认为 **几乎无实质修改**（保留了 GitHub 和 主题按钮）。
  - *注：过程中曾短暂隐藏了按钮，但最终根据您的需求恢复了。*

- **为什么要修改？**
  - 确保代码结构清晰，虽无功能变更，但确认了在受限模式下这些外链按钮是允许存在的。

- **如果不修改会怎样？**
  - 代码会变得臃肿，如果不封装，每次要判断状态都要写一遍 JWT 解析代码。

## 4. Hook (逻辑复用)

**文件路径**: `src/lib/hooks/use-recovery-mode.ts` (新文件)

- **做了什么修改？**
  - **封装逻辑**：将复杂的“JWT 解码判断 recovery 状态”的逻辑封装成一个 React Hook。

- **为什么要修改？**
  - **代码复用**：`AuthDialogManager` 和其他潜在组件都需要用到这个判断逻辑，避免代码重复。

- **如果不修改会怎样？**
  - 代码会变得臃肿，如果不封装，每次要判断状态都要写一遍 JWT 解析代码。

## 5. Layout (全局配置)

**文件路径**: `src/app/layout.tsx`

- **做了什么修改？**
  - **Metadata 修正**：将 `viewport` 配置从 `metadata` 对象中提取出来，单独导出。

- **为什么要修改？**
  - **消除警告**：Next.js 新版本废弃了在 `metadata` 中写 `viewport` 的做法，控制台会一直报黄字警告。

- **如果不修改会怎样？**
  - 功能正常，但开发时控制台会被警告刷屏，影响调试其他问题。
